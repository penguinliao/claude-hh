# 测试角色分工与协作原则

> 合并自：feedback_frontend_testing_gap / feedback_test_workflow_efficiency / feedback_user_perspective_testing / feedback_zhuolong_testing_principles / feedback_agent_self_capability

## 1. 小测与浊龙的职责分工（v2）

**小测 = 内部审计员（白盒）**
- API契约验证（curl每个端点，验证请求/响应格式）
- DB状态验证（操作后查表，验证数据正确）
- 权限矩阵验证（不同角色token，验证鉴权）
- 错误处理验证（故意传错参，验证友好错误）
- 性能基线验证（记录API响应时间）

**浊龙 = 用户体验验收员（黑盒）**
- 所有UI验证（页面渲染、按钮点击、流程走通）
- 用户操作流程（注册→使用→结果可见）
- AI回复质量评判
- 前端逐页逐按钮检查
- 小程序/App真机操作（screen-tap坐标点击）

> **v2变更**：UI测试不再由小测负责，全部交给浊龙。小测之前的"前端验收"职责（逐页检查bindtap、截图验证渲染）现在是浊龙的工作。小测专注API层和数据层的白盒审计。

## 2. 测试流程两轮收敛

测试清单必须一步到位覆盖所有角色×所有流程，不要先出34项再补到82项。

理想节奏：
1. 按角色×功能矩阵一次生成完整清单（每角色至少15-20项）
2. 跑一轮全量测试
3. 所有Bug批量修完
4. 跑一轮回归
5. 完（两轮收敛）

教训：聚伟测试跑了4轮才收敛（34→82→补测→回归），每轮浊龙25-45分钟，用户明确批评"测这么久"。

关键：浊龙启动前确认测试数据（账号、资源、订单）已就绪，避免中途补数据再跑一轮。

## 3. 验证必须用用户视角

测试验证必须用用户能做的事来验证，不靠查API/数据库。

例如验证"AI记住了吗"：
- ❌ 调API查数据库有没有 → 开发者视角
- ✅ 再问AI"我叫什么" → AI回答"张三" → 用户视角

浊龙的判定标准全部基于用户能感知的东西：
- **看到的** → 截图证明
- **AI回复的** → 对话验证
- **刷新后还在的** → 回访验证

api-check作为开发者模式保留，但：
- 默认测试流程不依赖它
- 不作为PASS/FAIL判定标准
- 只在开发者主动要求时使用，作为报告附录

## 4. 浊龙的两条核心原则

**原则1：screen-tap主力，绕过UI = 不是测试**
- automator直接调用小程序函数不是真实用户操作
- 如果用户点不到某个按钮，这本身就是P0 bug，浊龙应该报告而不是绕过
- automator只能用于环境准备（导航、设置数据），不能用于被测操作
- 所有screen-tap失败标记BLOCKED/FAIL报告，不用automator替代

教训：小鲲测试中输入框点不到（默认语音模式），浊龙用automator绕过去了，漏掉了P1 bug。

**原则2：测试方案必须预先制定，不能边测边想**
- 正确流程：收到交付单 → 侦察截图 → 制定详细方案（含覆盖率检查）→ 严格执行 → 覆盖率复查
- 错误流程：收到交付单 → 直接开测 → 测到哪算哪

教训：小鲲交付单10个场景，浊龙只测了4个（覆盖率40%），前面浪费31张截图在"找输入框"。

## 5. Agent能力必须内建，主Agent只做审核

设计子Agent时，所有能力必须长在子Agent自己身上。主Agent是临时教练——审核子Agent的自检是否到位，帮它校正，直到它能独立通过。

- ❌ 主Agent准备brief → 派子Agent执行 → 主Agent验收
- ✅ 子Agent自己生成brief → 主Agent审核brief → 子Agent执行+自验收 → 主Agent抽检

具体案例（浊龙）：
- **测试方案**：浊龙自己从交付单生成，不依赖主Agent
- **覆盖率检查**：浊龙自己做，主Agent抽检是否诚实
- **可信度判定**：浊龙自评，主Agent审核是否偏乐观
- **brief输出**：浊龙自己输出，未来CLI直接用

设计原则：问"如果没有主Agent，这个Agent能独立完成吗？"不能的话，把能力移到子Agent内部。主Agent只做校验层。
