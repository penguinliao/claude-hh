# 测试哲学与原则

> 合并自：feedback_test_strategy / feedback_test_thoroughness / feedback_fulltest_not_partial / feedback_test_data_isolation / feedback_ai_judgment_not_rules

## 1. 仿真测试优先，不跑pytest

用户是PM不懂代码，pytest结果看不懂。AI系统的核心质量在"对话是否正常"，不在"函数签名是否对"。

执行方式：
- 每个改动对应一个"用户会怎么用"的场景
- Python脚本直接调 `process_message()` 或核心函数，不启动HTTP
- Mock策略：AI调用用mock（返回预设JSON），员工服务用mock（返回预设结果）
- 检查3层：函数返回值 → 数据库状态 → 日志输出
- 纯逻辑验证（如JSON解析、CRUD）不需要AI，直接调函数
- AI密集型测试（调研/合同/方案）timeout设120-300秒

## 2. 全量覆盖，不只测改动

用户说"测试"默认是**全量测试每个项目的每个功能点**，除非明确说"只回归XX"。

原因：2026-04-03会话中，第一次只测记忆优化的17项，用户要求全量。重新跑231项后发现大量与改动无关的Bug（非法合同无拦截、多笔记账合并、并发阻塞）。

- 每个项目的每个action/API都要覆盖
- 不怕花时间花token——全量测试发现的Bug远多于只测改动
- 分组并行跑效率最高
- 按P0/P1/P2优先级执行：P0是"坏了用户看到灾难"（JSON泄露、误委派），P1是"影响体验"，P2是"有降级方案"

## 3. 实际运行验证，不读代码说PASS

测试不能只跑API或只看首页截图就说"通过"。必须：
- 逐个可交互元素实际操作（按钮、输入框、tab、链接）
- 检查每次操作后的状态变化（跳转、弹窗、toast、数据更新）
- 表单要实际填写+提交，不能只看空表单
- 修完Bug后必须回归测试

教训：管理后台侧边栏完全点不动、缺认证、测试数据未清理，之前的"验收"只截了静态页面。

## 4. 测试数据必须隔离

所有测试用独立账户（dev_前缀），绝不在真实用户账户上操作：
- 仿真测试用 `dev_simtest`、`dev_memverify`、`dev_audit_` 等前缀
- 自动检验也用测试账户发curl模拟，不让用户手动配合
- 对真实用户数据只做只读检查，不写入
- 测试结束必须清理：`delete_all_user_data('dev_xxx')`
- 能自动化的验证就不要让用户手动做
- 全量测试报告统一放 `~/Desktop/test_reports/`

## 5. AI用理解力判断，不硬编码规则

AI应用的判断逻辑靠AI理解，不靠规则匹配。给每种场景定制判定规则是打补丁思维，永远补不完。

- ❌ 委派测试看到"已安排"文字 → PASS（硬编码关键词）
- ✅ AI理解意图"用户要拿到文章"，当前只看到"已安排" → 还没完成，继续等

推广原则：
- 任务完成判定 → AI理解"用户需求满足了吗？"
- 回复质量判定 → AI理解意图后评分，不是关键词检查
- 监控告警 → AI理解"这个状态正常吗？"，不是阈值触发
- 只有确定性判断（非空检查、格式校验）才用规则

## 对抗性测试（必做刁钻场景）

每次测试必须包含这些场景：
- 空字符串、超长输入（500字+）、特殊字符（<>&"'emoji）
- 同一按钮连点3次、同一数据创建3次
- 幂等性：同一操作执行两次会怎样
- 空数据冷启动：全新用户/空项目不报错不白屏
- 降级路径：API超时/500时前端表现
- 数据一致性：前端显示 == 数据库实际数据
- 同类问题排查：发现一个bug后全局搜索同类问题
