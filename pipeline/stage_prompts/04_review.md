# Stage 4: REVIEW

## 核心原则：写代码的和查代码的不能是同一个上下文

Opus 在 Stage 3 调度了 Sonnet 写代码。现在 Opus 用**全新的审查 agent**来审查代码——这个 agent 不知道代码是怎么写的，只看结果。

### 为什么要用新 agent？

同一个上下文窗口 = 同样的盲区。开一个新窗口总能发现新问题，因为上下文污染被清除了。

### Opus 的工作

派一个独立的审查 agent，prompt 里**只给文件路径和验收标准**，不给 Stage 3 的任何上下文：

```python
Agent(
    subagent_type="general-purpose",
    model="sonnet",  # 审查不需要 Opus，Sonnet 够用
    prompt="""你是独立代码审查员，不知道这段代码是怎么写的。

## 流水线规则（必须遵守）

1. **你是审查员，不是开发者**：只读代码、只记录问题，绝对不修改任何代码文件
2. **不修改 pipeline.json**：不要运行 advance/retreat/reset，流水线控制是 Opus 的事
3. **发现问题只记录**：输出审查报告，修复是 Stage 3 的事

## 审查任务

请审查以下文件：[修改的文件列表]
项目根目录：[project_root 的绝对路径]
对照验收标准：[.harness/spec.md 内容]

逐项检查：
1. 功能完整性：spec 每条验收标准都有对应实现
2. 安全性：SQL参数化、无硬编码密钥、日志脱敏、权限检查
3. 健壮性：外部调用有超时和错误处理、无资源泄露
4. Import完整性：所有import的模块/函数真实存在
5. 跨文件签名一致：调用参数和被调函数签名匹配
6. 返回值一致性：同一函数所有return路径类型一致

每项标注 PASS 或 FAIL，FAIL 必须给出文件:行号和具体问题。
不接受"看起来不错"——必须有证据。

输出格式：
| # | 检查项 | 结果 | 证据 |
|---|--------|------|------|

汇总：X/6 通过。未通过项列出具体修复建议。""",
)
```

### 审查结果处理

- **全部 PASS** → advance 到 Stage 5
- **有 FAIL** → retreat 回 Stage 3，Opus 派 Sonnet 修复具体问题
- **REVIEW→IMPLEMENT→REVIEW 最多循环 3 次**，3 次后停下报告 PM

### 绝对不做

- **不改代码**：发现问题只记录，修复回 Stage 3
- **不在当前上下文审查**：必须派新 agent，保证认知隔离
- **不放水**：不因为"代码是我的 Sonnet 写的"就降低标准
