# 规格先行工作流详解

## 为什么规格先行

### 研究支撑：Self-Spec（ICLR 2026）

Self-Spec 研究表明，让 AI 在编码前先输出一份规格文档，再按规格实现，pass 率提升 2-5%。这个数字看起来不大，但要注意：

- 这是在已经很强的基线上的提升（Claude Sonnet 在 SWE-bench 上已达 70%+）
- 提升来源不是"写更好的代码"，而是"更准确地理解需求"
- 在复杂任务上提升更显著（多文件修改场景提升可达 8-12%）

### 核心洞察：瓶颈已经转移

AI 编码的失败原因分布（2026 年数据）：

| 失败原因 | 占比 | 趋势 |
|----------|------|------|
| 理解需求偏差 | 45-55% | 稳定，难以靠模型能力解决 |
| 实现逻辑错误 | 20-25% | 随模型升级持续下降 |
| 环境/工具问题 | 15-20% | 工程化可解决 |
| 边界条件遗漏 | 10-15% | 规格化可解决 |

**关键事实**：当任务定义清晰（有明确的输入、输出、边界条件）时，AI 的实现失败率已降到 0-1%。瓶颈不再是"AI 不会写代码"，而是"AI 不确定你要什么"。

规格先行的本质：**把模糊的人类意图，转化为清晰的工程定义，再交给 AI 执行**。

## 完整工作流

```
需求（自然语言）
    │
    ▼
┌─────────────────────┐
│ Step 1: AI 输出规格  │  AI 把需求翻译成结构化规格
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│ Step 2: 人确认规格   │  人检查"AI 理解的对不对"（不是检查代码）
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│ Step 3: 模块化实现   │  按规格逐模块实现，每个模块单步聚焦
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│ Step 4: 自测验证     │  按验收标准逐条验证
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│ Step 5: 集成测试     │  模块组装后的端到端验证
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│ Step 6: 人审查交付   │  人确认功能效果，不是看代码
└─────────────────────┘
```

### Step 1: AI 输出规格

给 AI 的 prompt 模板：

```
我需要[功能描述]。

请先输出一份规格文档，包括：
1. 你对需求的理解（用你自己的话复述）
2. 验收标准（"当X时，应该Y"格式）
3. 边界条件和异常场景
4. 需要修改的文件和修改内容
5. 依赖和约束

不要写代码，只输出规格。
```

### Step 2: 人确认规格

人只需要回答三个问题：
- **理解对不对？** AI 复述的需求和你想要的一样吗？
- **场景全不全？** 有没有遗漏的边界条件？
- **范围对不对？** 修改的文件范围合理吗？

确认方式：
- 完全正确 → "开始"
- 需要调整 → "第3条验收标准改成..."
- 理解偏差大 → "你理解错了，我要的是..."

### Step 3-6: 按单步聚焦原则执行

详见 [single-focus-principle.md](./single-focus-principle.md)。

## 规格文档模板

完整模板见 [spec-template.md](../templates/spec-template.md)。

### 示例：为博客系统添加标签功能

```markdown
# 博客标签功能 规格文档

## 需求理解
我理解的需求是：为博客系统添加标签（Tag）功能，用户可以给文章打标签，
读者可以按标签筛选文章。标签是扁平结构（不是树形分类），一篇文章可以
有多个标签。

## 验收标准
1. 当用户创建/编辑文章时，应该能添加1-10个标签
2. 当用户输入标签时，应该自动补全已有标签
3. 当读者点击标签时，应该看到该标签下的所有文章（按时间倒序）
4. 当文章被删除时，应该自动清理标签关联（标签本身保留）
5. 当标签下没有文章时，应该在标签列表中隐藏该标签

## 边界条件和异常场景
- 当用户输入超过10个标签时，应该提示"最多10个标签"并阻止提交
- 当标签名超过20个字符时，应该截断并提示
- 当标签名包含特殊字符时，应该只保留中文、英文、数字和连字符
- 当两个标签仅大小写不同时，应该视为同一标签（统一小写存储）

## 修改范围
| 文件 | 修改内容 | 风险点 |
|------|---------|--------|
| models/tag.py | 新增Tag模型和ArticleTag关联表 | 数据库迁移 |
| models/article.py | 添加tags关系 | 无 |
| api/articles.py | 创建/编辑接口支持标签参数 | 需兼容旧接口 |
| api/tags.py | 新增标签列表和搜索接口 | 无 |
| templates/editor.html | 添加标签输入组件 | 前端交互复杂度 |
| templates/article.html | 显示标签和标签链接 | 无 |

## 依赖和约束
- 数据库需要新增两张表，需执行迁移
- 标签自动补全需要一个搜索接口，响应时间应<200ms
- 不引入新的前端框架，用现有的Alpine.js实现
```

## 验收标准写法

### 格式："当X时，应该Y"

好的验收标准具备三个特征：
1. **可观测**：人或自动化工具能验证
2. **无歧义**：只有一种理解方式
3. **独立**：每条单独可验证

### 好的例子

```
当用户上传大于5MB的图片时，应该自动压缩到5MB以下并显示"已自动压缩"提示
当网络断开时，应该将未发送的消息存入本地队列，恢复后自动重发
当用户连续3次输入错误密码时，应该锁定账户15分钟并发送邮件通知
```

### 坏的例子

```
# 太模糊
系统应该性能良好  → 改为：首页加载时间应该<2秒（P95）

# 不可验证
用户体验应该流畅  → 改为：当用户点击按钮后，应该在200ms内显示loading状态

# 多个条件混在一起
用户应该能管理标签，包括添加、删除和编辑  → 拆成3条独立的验收标准
```

## 与同类工具对比

| 维度 | 规格先行（本框架） | GitHub Spec Kit | AWS Kiro |
|------|-------------------|-----------------|----------|
| **定位** | 通用方法论 + 模板 | GitHub 集成工具 | IDE 插件 |
| **规格生成** | AI 对话生成 | 从 Issue 自动生成 | 从需求文档生成 |
| **规格格式** | Markdown（灵活） | YAML（结构化） | Kiro 专有格式 |
| **验证方式** | 人确认 + 自测脚本 | CI/CD 集成 | IDE 内实时验证 |
| **锁定程度** | 无锁定 | GitHub 生态 | AWS 生态 |
| **适用场景** | 任何 AI 编码场景 | GitHub 项目管理 | 大型企业项目 |
| **学习成本** | 低（只是写文档） | 中（需学工具链） | 高（需学 Kiro 体系） |

**推荐策略**：
- 小团队/个人项目 → 本框架（零成本，灵活）
- GitHub 重度用户 → GitHub Spec Kit + 本框架的验收标准写法
- 企业级项目 → AWS Kiro 或自建（本框架的思想仍然适用）

## 快捷模式：不同规模需要多少规格化

不是每个任务都需要完整的规格文档。根据任务规模选择合适的程度：

### Level 0: 零规格（直接做）

**适用**：改文案、调样式、修 typo 等确定性极高的小改动

```
把首页标题改成"欢迎回来"
```

### Level 1: 一句话规格（10秒）

**适用**：1-2个文件的简单功能修改

```
给用户列表加个"最后登录时间"列，数据从user表的last_login字段读取，格式"X天前"
```

### Level 2: 验收标准（2分钟）

**适用**：涉及 3-5 个文件的中等功能

```
需求：添加用户导出功能

验收标准：
1. 当管理员点击"导出"时，应该下载CSV文件
2. CSV应该包含：用户名、邮箱、注册时间、最后登录
3. 当用户超过10000时，应该后台生成并邮件通知下载链接
```

### Level 3: 完整规格（10-15分钟）

**适用**：涉及 5+ 个文件的复杂功能，或多人协作

使用完整的 [spec-template.md](../templates/spec-template.md)。

### Level 4: 规格 + ADR（30分钟）

**适用**：涉及架构决策的大型变更

在完整规格之外，额外写 [ADR](../templates/adr-template.md) 记录架构决策。

### 选择指南速查

| 改动文件数 | 是否涉及新概念 | 是否多人协作 | 推荐 Level |
|-----------|---------------|-------------|-----------|
| 1 | 否 | 否 | 0 |
| 1-2 | 否 | 否 | 1 |
| 3-5 | 可能 | 否 | 2 |
| 5+ | 是 | 否 | 3 |
| 5+ | 是 | 是 | 4 |
