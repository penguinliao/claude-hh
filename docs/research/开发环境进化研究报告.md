## 一、你的问题到底出在哪？——用RL框架做根因分析

### 1.1 一句话诊断

> **AI写代码的语法正确率已达95%，但安全通过率三年来稳定在55%。你的367个Bug不是AI"写错了"，而是没有环境告诉它什么是对的。**

### 1.2 RL视角的根因分析

强化学习（RL）的核心公式：**Agent在环境中行动，环境给奖励信号，Agent根据奖励调整行为**。

把你的开发过程映射到这个框架：

| RL概念 | 你的开发中 | 当前状态 | 问题 |
|--------|-----------|---------|------|
| **Agent** | Claude Code | 很强（Opus 4.6） | 模型本身没问题 |
| **状态** | 代码库 | 8个项目，300+文件 | 项目多但各自独立 |
| **动作** | 编辑/创建代码 | AI在写代码 | 写得快但没检查 |
| **奖励信号** | ？？？ | **几乎为零** | **这是根因** |
| **环境** | 开发工具链 | 只有py_compile | **极度简陋** |

**类比理解**：你现在的状态相当于让一个天才厨师在没有味觉的情况下做菜——他什么都会做，但做完了不知道咸淡。你需要的不是换厨师（换模型），而是**给他一套味觉系统**（工程化环境）。

## 二、行业正在发生什么？——四个时代的演进

### 2.1 从Vibe Coding到Harness Engineering

Andrej Karpathy在2025年2月提出"Vibe Coding"，到2026年初他自己宣布这个概念已经"过时"。行业在一年内经历了四个阶段：

```
2023-2024  Vibe Coding（氛围编程）
           "写个prompt然后祈祷"
           ↓ 你之前在这个阶段

2024-2025  Plan Coding（计划编程）
           "给AI一个分步计划"
           ↓ 你的CLAUDE.md已经到了这个阶段

2025       Spec Coding（规格编程）
           "给AI详细规格+验收标准"
           ↓ 你的仿真测试和浊龙是这个阶段的实践

2026.02    Harness Engineering（驾驭工程）← 当前前沿
           "给AI一个有自动反馈回路的完整环境"
           → 这是你需要进化到的阶段
```

### 2.2 Harness Engineering的正式定义

**2026年2月**，三个重量级人物/组织同时定义了这个概念：

**Mitchell Hashimoto**（HashiCorp/Terraform创始人）：
> "任何时候你发现Agent犯了一个错误，你就花时间工程化一个解决方案，让Agent永远不再犯同样的错误。"

**OpenAI**（官方实践报告）：
> 他们有一个**超过100万行代码的产品，零行人写代码**。工程师不写代码——他们设计让AI可靠写代码的系统。

**Martin Fowler**（软件工程教父）：
> Harness = Guides（前馈控制，行动前引导） + Sensors（反馈控制，行动后检查）

### 2.3 核心公式

> **Agent = Model + Harness**

Model（模型）你已经有了最好的（Claude Opus 4.6）。你缺的是Harness（驾驭环境）。

## 五、RL强化学习：不需要训练模型，只需要搭环境

### 5.1 核心洞察

> **你不需要懂RL算法，但RL的思维框架是搭建Harness的理论基础。**

RL的核心循环：

```
Agent观察状态 → 采取行动 → 环境给奖励 → Agent调整策略
     ↓              ↓           ↓              ↓
AI读代码库    →  编辑代码   → 工具检查结果  → AI修复问题
```

**类比**：教一个实习生写代码，你不需要"重新训练他的大脑"（那是大学的事），但你可以：
1. **设定明确的验收标准** = 奖励函数（lint通过、安全扫描通过、测试通过）
2. **提供即时反馈工具** = 环境（ruff、bandit、mypy、pytest）
3. **允许多次迭代** = 多步决策（写→检查→改→再检查→直到通过）

### 5.2 多维奖励函数设计

```
代码质量奖励（概念版）：

所有测试通过：        +10分
没有安全漏洞：        +8分（Bandit/Semgrep）
类型检查通过：        +5分（mypy）
没有lint警告：        +3分（Ruff）
没有密钥泄露：        +3分（detect-secrets）
代码覆盖率>80%：     +2分
没有性能退化：        +2分
引入新依赖：          -1分
增加代码复杂度：      -1分
```

**每一项都对应一个自动化工具**，不需要人工判断。

### 5.3 验证者vs生成者——最实用的RL概念

> **验证一个答案是否正确，通常比生成一个正确答案容易得多。**

- 检查代码有没有SQL注入：自动工具几秒搞定（验证）
- 写出没有SQL注入的代码：需要深入理解业务逻辑（生成）

**应用方式**：与其让AI"一次写对"，不如让它写完后自动跑验证，验证不过就自动修复再跑。**验证工具越强，生成质量越高。**

### 5.4 你已经在做的"RL式"实践

| 你的实践 | RL对应 |
|---------|--------|
| CLAUDE.md规则 | 策略的先验知识（前馈引导） |
| 仿真测试 | 奖励函数（行为验证） |
| 小测（白盒） | 内部状态奖励信号 |
| 浊龙（黑盒） | 外部行为奖励信号 |
| 跨会话memory更新 | 长期策略更新 |

**缺失的环节**：代码写完到提交之间的**即时反馈**（内循环）——这是本报告的核心建议。

### 6.1 总体架构：五层防护

```
┌─────────────────────────────────────────────────────┐
│  第5层：黑盒验收（浊龙）                              │
│  → 用户视角端到端测试                                 │
├─────────────────────────────────────────────────────┤
│  第4层：白盒测试（小测）                              │
│  → 仿真场景 + 逻辑验证                               │
├─────────────────────────────────────────────────────┤
│  第3层：CI质量门禁（GitHub Actions）                  │  ← 你需要新增
│  → PR级全量检查，不过不让合并                         │
├─────────────────────────────────────────────────────┤
│  第2层：内循环即时检查（Hooks + Pre-commit）          │  ← 你需要新增
│  → 代码写完秒级反馈：lint、类型、安全、密钥           │
├─────────────────────────────────────────────────────┤
│  第1层：前馈引导（CLAUDE.md + 安全规则 + 架构文档）    │  ← 你需要加强
│  → AI动手前就知道什么能做什么不能做                    │
└─────────────────────────────────────────────────────┘
```

你现在有第4、5层（测试），但**第1-3层几乎为空**。这就是为什么Bug能一路穿透到生产环境。

### 6.2 第1层：前馈引导（Guides）

#### 6.2.1 安全编码规则（零成本、立即生效）

**Wiz安全规则文件**（github.com/wiz-sec-public/secure-rules-files）：
- 开源，覆盖Python/Flask/Django
- 直接加入CLAUDE.md即可
- 包含：禁止硬编码密钥、要求参数化查询、禁用eval()等

**OpenSSF AI安全编码指南**（best.openssf.org）：
- 最权威的AI编码助手安全指令
- 关键发现：**不要告诉AI"你是安全专家"**（反而降低效果），要给具体可操作的规则
- 推荐**递归批判与改进（RCI）**：让AI检查自己的输出找安全问题

#### 6.2.2 架构决策记录（ADR）

每次AI做架构选择时（选框架、设计数据模型、选通信方式），用标准格式记录：
- 什么问题需要解决
- 考虑了哪些方案
- 选择了什么，为什么
- 风险和后果

**在CLAUDE.md中加入**："每次做架构决策时，先写一个ADR让我确认。"

#### 6.2.3 AGENTS.md标准

2026年由Linux Foundation管理的新标准，Sourcegraph/OpenAI/Google/Cursor联合推动。工具无关的配置文件，所有AI Agent都能读。

建议：维护一个权威AGENTS.md，通过脚本自动生成CLAUDE.md等各工具格式。

### 6.3 第2层：内循环即时检查（Sensors）

#### 6.3.1 工具链配置

| 工具 | 用途 | 速度 | 能拦截的你的Bug类型 |
|------|------|------|-------------------|
| **Ruff** | 代码风格+基础安全 | <1秒/万行 | 死代码、格式问题、基础错误 |
| **mypy --strict** | 类型检查 | ~5秒 | `.strip()`对None、format()崩溃、Optional未处理 |
| **Bandit** | Python安全扫描 | ~3秒 | SQL注入、硬编码密钥、不安全随机数 |
| **detect-secrets** | 密钥泄露检测 | <1秒 | API Key明文泄露 |
| **Semgrep** | 深度安全扫描 | ~10秒 | XSS、路径穿越、XXE、认证绕过 |

**总耗时 < 20秒**，完全不影响AI迭代速度。

#### 6.3.2 Pre-commit配置

```yaml
# .pre-commit-config.yaml（所有项目统一使用）
repos:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.6.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
      - id: check-added-large-files
      - id: detect-private-key        # 检测私钥文件

  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.8.0
    hooks:
      - id: ruff                       # lint检查
        args: [--fix]
      - id: ruff-format               # 自动格式化

  - repo: https://github.com/PyCQA/bandit
    rev: 1.8.0
    hooks:
      - id: bandit                     # 安全扫描
        args: [-r, --severity-level, medium]

  - repo: https://github.com/Yelp/detect-secrets
    rev: v1.5.0
    hooks:
      - id: detect-secrets             # 密钥泄露检测

  - repo: https://github.com/pre-commit/mirrors-mypy
    rev: v1.13.0
    hooks:
      - id: mypy                       # 类型检查
        args: [--strict]
```

#### 6.3.3 Claude Code Hooks配置

在Claude Code的settings中配置：
- **PostToolUse（文件写入后）**：自动运行ruff + bandit扫描刚修改的文件
- **PreCommit**：提交前运行完整安全扫描链

### 6.4 第3层：CI质量门禁

```yaml
# .github/workflows/quality-gate.yml
name: AI Code Quality Gate
on: [push, pull_request]

jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.12' }

      - name: Ruff lint + format check
        run: pip install ruff && ruff check . && ruff format --check .

      - name: Type check (mypy)
        run: pip install mypy && mypy --strict src/

      - name: Security scan (Bandit)
        run: pip install bandit && bandit -r src/ --severity-level medium

      - name: Dependency vulnerability check
        run: pip install safety && safety check

      - name: Secret scan
        uses: trufflesecurity/trufflehog@main
```

### 6.5 共享基础设施：解决跨项目重复Bug

#### 问题
你的8个项目重复出现相同Bug：DB连接泄漏、JWT默认密钥、CORS配置错误。因为AI每次都从头"重新发明"这些基础组件。

#### 方案：共享Python包

创建 `starpalace-shared` 包：

```
starpalace-shared/
├── starpalace_shared/
│   ├── db.py          # 统一安全连接池（再也不用手动close）
│   ├── auth.py        # 统一认证中间件（JWT密钥必须配置）
│   ├── security.py    # 统一CORS/安全头（禁止*通配符）
│   ├── models.py      # 统一Pydantic基础模型（strict模式）
│   └── logging.py     # 统一日志配置（禁止print）
├── pyproject.toml
└── tests/
```

每个项目只需：`pip install starpalace-shared`

**效果**：AI不再需要每次都写连接池/认证/CORS，直接import安全的版本。跨项目重复Bug从此消失。

#### 项目模板

创建标准模板仓库，新项目从此模板创建：

```
project-template/
├── .pre-commit-config.yaml    # 预配置安全扫描链
├── pyproject.toml              # ruff + mypy + bandit配置
├── CLAUDE.md                   # 安全编码规则 + 项目约定
├── src/
│   ├── core/                   # 引用starpalace-shared
│   └── api/                    # FastAPI标准结构
├── tests/
└── .github/workflows/
    └── quality-gate.yml        # CI质量门禁
```

| 注入攻击 | 复杂查询用字符串拼接 | XXE攻击、XSS |
| 不安全设计 | 不会"推回"不安全的设计 | 路径穿越 |
| 安全配置错误 | CORS用*通配符 | `allow_origins=*` |

### 7.3 好消息

> **加装pre-commit安全扫描链后，研究表明可以在提交前拦截92%的Python安全漏洞。**

## 八、三层开发者循环框架（Gene Kim + Steve Yegge）

### Inner Loop（秒到分钟）——实时人机协作

- 每次AI修改代码后，自动跑lint + 类型检查 + 安全扫描
- 失败信息直接反馈给AI → 自动修复 → 再跑 → 直到通过
- **频率**：每次文件修改都触发

### Middle Loop（小时到天）——跨会话连续性

- 维护CLAUDE.md/AGENTS.md规则文件
- 每次发现新问题 → 编码为规则 → 加入规则文件 → AI未来不再犯
- 架构决策记录（ADR）防止架构漂移
- **这就是Harness的自进化**

### Outer Loop（周到月）——架构和组织

- 共享基础设施（starpalace-shared包）
- CI/CD门禁
- 定期安全审计（用Semgrep深度扫描）
- 项目模板更新

## 十一、一句话总结

> **你不需要换AI模型，不需要学编程，不需要懂RL算法。你需要的是给AI装上"自动安检门"——代码写出来的瞬间，lint检查格式、mypy检查类型、bandit检查安全、detect-secrets检查密钥，不过关就自动打回让AI修，过关了才允许提交。这就是Harness Engineering的本质：把367个Bug的教训，编码成永远不会遗忘的自动化规则。**
