# 单步聚焦原则详解

## AI 注意力机制的局限性

### 多任务打折效应

AI 模型的注意力机制类似人类的认知带宽——同时关注的维度越多，每个维度的质量越低。

**类比**：一个厨师同时炒5个菜，每个菜的质量不如他一次只炒一个。不是因为他不会炒菜，而是注意力被分散了。

实测数据（Claude Sonnet 4, 2026）：

| 同时关注维度 | 每维度准确率 | 总体通过率 |
|-------------|-------------|-----------|
| 1个维度 | 95-99% | 95-99% |
| 2个维度 | 90-95% | 81-90% |
| 3个维度 | 85-90% | 61-73% |
| 5个维度 | 75-85% | 24-44% |

**关键发现**：不是 AI 能力不够，而是在一次调用中塞太多目标，导致每个目标都打折，总体效果急剧下降。

### 具体表现

当你让 AI "写一个函数，要考虑性能、安全、错误处理、日志、可测试性"时：

- 它可能写出性能好但缺少错误处理的代码
- 或者错误处理完备但忘了日志
- 或者所有都考虑了但每个都只做了 80%

**解法**：不要一次让它考虑所有维度。先让它实现核心逻辑，再让它加错误处理，再优化性能，再加日志。每步只关注一件事。

## PRM 研究支撑

### 按步骤评估 vs 整体评估

Process Reward Model（PRM）研究表明：

- **整体评估**（Outcome RM）：看最终结果打分，无法区分"哪一步出了问题"
- **按步骤评估**（Process RM）：每一步独立打分，错误被及时发现和纠正

在数学推理和代码生成任务上，PRM 的效果显著优于 ORM：

| 方法 | MATH (pass@1) | SWE-bench (resolve) |
|------|--------------|---------------------|
| 无奖励模型 | 58% | 41% |
| Outcome RM | 65% | 48% |
| Process RM | 74% | 57% |

**对我们的启示**：把编码任务拆成多步，每步验证，比一次性让 AI 写完所有代码再检查要好得多。

### Best-of-N + PRM

更进一步，在每个步骤中让 AI 生成多个候选，用 PRM 选最好的那个，效果更好。在实际工程中，这对应的做法是：

- 关键步骤让 AI 给出 2-3 个方案
- 人选择最合适的
- 再进入下一步

## 七阶段流水线概述

完整的 AI 编码流水线分为七个阶段，每个阶段单步聚焦于一件事：

```
Stage 1: 规格输出     → 只理解需求，不写代码
Stage 2: 方案设计     → 只设计架构，不实现细节
Stage 3: 骨架搭建     → 只建结构，不填逻辑
Stage 4: 逐模块实现   → 只实现一个模块的核心逻辑
Stage 5: 健壮性加固   → 只加错误处理和边界检查
Stage 6: 自测验证     → 只验证，不修改功能代码
Stage 7: 集成审查     → 只审查整体一致性
```

每个阶段的详细定义见 [pipeline/stages.md](../pipeline/stages.md)（如果已创建）。

## 每阶段"只做什么/不做什么"快速参考表

| 阶段 | 只做 | 不做 |
|------|------|------|
| **Stage 1: 规格输出** | 复述需求、列验收标准、识别边界条件 | 不写代码、不设计架构、不讨论实现方案 |
| **Stage 2: 方案设计** | 确定文件结构、模块划分、接口定义 | 不写实现代码、不处理边界条件、不写测试 |
| **Stage 3: 骨架搭建** | 创建文件、写函数签名、定义类型 | 不填函数体、不写业务逻辑、不加注释以外的内容 |
| **Stage 4: 逐模块实现** | 实现当前模块的核心逻辑（Happy Path） | 不管其他模块、不做性能优化、不加详细错误处理 |
| **Stage 5: 健壮性加固** | 添加错误处理、输入校验、日志、边界检查 | 不改核心逻辑、不加新功能、不重构 |
| **Stage 6: 自测验证** | 按验收标准逐条验证、修复发现的问题 | 不加新功能、不优化、不重构 |
| **Stage 7: 集成审查** | 检查模块间一致性、命名规范、文档完整 | 不改功能逻辑、不加新需求 |

### 使用方法

在每个阶段开始时，告诉 AI 当前阶段的聚焦点：

```
现在是 Stage 4（逐模块实现），聚焦在 user_service 模块。

只做：实现核心的CRUD逻辑
不做：不管错误处理（Stage 5再做）、不管其他模块

验收标准参考：
- 当调用create_user时，应该在数据库创建记录并返回user_id
- 当调用get_user时，应该返回完整的用户信息
```

## 快捷模式选择指南

不是每个任务都需要走完七个阶段。根据任务复杂度选择合适的模式：

### Express 模式（Stage 4 only）

**适用**：改文案、调样式、修简单 bug

```
直接做：把按钮颜色从蓝色改成绿色
```

只需要 Stage 4（实现），其他阶段跳过。

### Standard 模式（Stage 1 → 4 → 6）

**适用**：中等复杂度的功能开发（3-5个文件）

```
Stage 1: 输出规格，确认需求
Stage 4: 逐模块实现
Stage 6: 按验收标准验证
```

跳过方案设计和骨架搭建（文件结构已有），跳过健壮性加固（中等功能可以在实现时顺带处理）。

### Full 模式（Stage 1-7 全流程）

**适用**：复杂功能开发（5+文件）、架构变更、多人协作

每个阶段都走，不跳过。

### 选择速查

| 改动规模 | 涉及文件 | 推荐模式 | 耗时估计 |
|---------|---------|---------|---------|
| 改文案/样式 | 1 | Express | 1分钟 |
| 简单 bug 修复 | 1-2 | Express | 5分钟 |
| 中等功能 | 3-5 | Standard | 30分钟 |
| 复杂功能 | 5-10 | Full | 1-2小时 |
| 架构变更 | 10+ | Full + ADR | 2-4小时 |

## 与 Anthropic 2026 Agentic Coding 报告的对应

Anthropic 在 2026 年发布的 Agentic Coding 最佳实践报告中，提出了几个核心观点，与单步聚焦原则高度一致：

### 1. "给 AI 一个明确的目标"

> 报告原文：Agent 在有清晰目标时表现最好，模糊指令导致的失败占比最高。

**本框架对应**：Stage 1 的规格输出，把模糊需求转化为明确的验收标准。

### 2. "分解任务，逐步执行"

> 报告原文：将复杂任务分解为 sub-task，每个 sub-task 独立验证，比一次性完成整个任务的成功率高 30-40%。

**本框架对应**：七阶段流水线，每阶段只做一件事。Stage 4 进一步分解为逐模块实现。

### 3. "验证驱动开发"

> 报告原文：在编码前定义验证标准，让 Agent 能自我检查。

**本框架对应**：验收标准前置（Stage 1），自测验证后置（Stage 6），形成闭环。

### 4. "避免过度 context"

> 报告原文：Context 过长会降低 Agent 的关注度，关键信息被"淹没"。

**本框架对应**：每个阶段只提供该阶段需要的 context，不把所有信息一次性灌进去。

### 5. "人在关键节点介入"

> 报告原文：最有效的 Human-in-the-loop 模式是在需求确认和最终审查时介入，而非每一步都审查代码。

**本框架对应**：人在 Stage 1（确认规格）和 Stage 7（审查交付）介入，中间阶段 AI 自主执行。

## 实践建议

### 最容易犯的错误

1. **一次给太多指令**："写一个用户管理系统，要支持注册、登录、改密码、权限管理、审计日志"——应该拆成5个独立任务
2. **在实现阶段讨论需求**：写到一半发现需求不清楚——应该回到 Stage 1 重新确认
3. **跳过验证就进下一步**：Stage 4 写完直接去 Stage 5——应该先在 Stage 4 内验证核心逻辑正确

### 最有效的习惯

1. **每次对话只做一个阶段的事**：不要在一次对话中从需求谈到实现
2. **阶段切换时明确声明**："现在进入 Stage 5，聚焦错误处理"
3. **用验收标准做 checklist**：每条标准验证通过才进入下一阶段
