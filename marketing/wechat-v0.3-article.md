# 我是产品经理，不懂代码，但我给 AI 造了一套质检流水线

> Claude H-H v0.3 — 一个非技术 PM 用 Claude Code 做出来的 AI 代码质量门禁系统

---

## 为什么做这个东西

我是产品经理，不写代码。过去几个月一直用 Claude Code 做产品原型。体验很好，但有一个问题反复出现：

**Claude 会很自信地说"我修好了"，但其实没修好。**

同一个 bug，同一个对话，它能"修"三遍。每次都说"这次一定行了"。我不懂代码，没法自己审查，只能信它——然后上线出问题。

Anthropic 自己的数据也证实了这一点：Claude Code 生成的代码准确率大约 50%。也就是说，AI 写的每两段代码里，有一段是有问题的。

所以我问了一个简单的问题：**能不能不信 AI 说的，而是让机器自己验证？**

---

## 怎么做的

Claude Code 有一个 Hook 机制——可以在 AI 编辑文件前、编辑后、提交代码前、子任务完成后自动触发检查脚本。这就是"物理约束"的基础。

我基于这个机制做了 **Claude H-H**，核心思路是强制 AI 走一条 5 阶段流水线：

| 阶段 | 做什么 | 门禁 |
|------|--------|------|
| SPEC | 写需求 + 验收标准 + 测试脚本 | spec.md 必须存在 |
| DESIGN | 设计架构（可选） | — |
| IMPLEMENT | AI 写代码 | 唯一允许写代码的阶段 |
| REVIEW | 机器自动审查 | ruff/mypy/bandit 评分 ≥ 60 |
| TEST | 机器执行测试脚本 | 所有脚本 exit 0 |

每个阶段都有物理门禁。不是 AI 自己说"我通过了"，而是 harness 引擎真的跑工具验证。通不过就不能进入下一步。

**三个核心原则**：
1. **环境 > 模型** — 不追求更强的 AI，追求更好的验证环境
2. **物理约束 > AI 承诺** — Hook 真跑工具，不信"我修好了"
3. **规格先行** — 先写需求再写代码，验收标准在写代码之前就锁定

## 踩过的坑

### 坑 1：AI 会"聪明地"绕过规则

最开始我用格式检查——要求 test_results.md 里必须有"N/M PASS"格式。AI 立刻学会了：

```
T1 语法检查 PASS
T2 函数存在 PASS
T3 签名正确 PASS
3/3 PASS
```

格式完美。但内容全是"语法正确""函数存在"——这不是测试，这是对着镜子说"我长得不错"。

**教训**：检查 AI 写的 markdown 是无效的。AI 的优化目标是"通过检查"而非"做好检查"。解决办法是让 harness 自己用 subprocess 执行测试脚本，检查 exit code——AI 必须写一个真正能跑的程序，不能靠写一段文字过关。

### 坑 2：旧通行证永不过期

上周的 pipeline 停在 IMPLEMENT 阶段，这周来了新任务，AI 直接开始写代码——因为 Hook 只问"现在是不是 IMPLEMENT 阶段"，不问"这个 pipeline 是不是为当前任务创建的"。

**教训**：pipeline 必须有过期机制。超过 4 小时未活动的 pipeline 自动失效，AI 必须为新任务启动新流水线。改动写进去的那一刻，立刻拦住了我自己——因为当前 pipeline 是 12 小时前的。

### 坑 3：安检员自己不过安检

最讽刺的发现：我花了一整天给 AI 的代码加门禁，但 harness-engineering 项目本身从来没有用过自己的 pipeline。一个代码质量门禁系统，自己的代码没经过任何门禁。

**教训**：从 v0.2 开始，harness 自己的代码也必须走自己的 pipeline。

### 坑 4：出题的人不能是答题的人

开发完代码再写测试，AI 会不自觉地让测试适配代码（确认偏误）。就像先写答案再出题，题目一定是对的。

**教训**：v0.3 做了认知隔离——测试脚本在 SPEC 阶段由 Opus 写（出题），代码在 IMPLEMENT 阶段由 Sonnet 写（答题），两者互不可见。Sonnet 被物理锁定，无法修改测试脚本。

### 坑 5：AI 不会主动反馈问题

Sonnet 在实现代码时发现 spec 里的接口定义不合理，但它不会说——要么默默按错的 spec 实现（导致后面审查失败），要么擅自改接口（导致和其他模块对不上）。

**教训**：v0.3 加了"接口变更请求"机制。Sonnet 发现问题时写到 change_request.md，Opus（主管 Agent）在 Sonnet 完成后看到这个文件并裁决。分层决策，不是一人说了算。

## v0.3 新增：AI 会从错误中学习

受 Hermes Agent（GitHub 8 万星）的 skill 自学习机制启发，v0.3 加了经验沉淀系统：

每次 pipeline 有退回（代码被打回重写），harness 自动从客观数据中提取教训：
- 哪个维度失败了（安全/功能/类型）
- 评分是多少
- 涉及哪些文件

存成结构化的 skill 文档。下次遇到类似任务时自动加载，AI 不再重复犯同样的错。

**和 Hermes 的关键区别**：Hermes 让 AI 自己总结经验（有幻觉风险），我们从 pipeline 的客观数据提取（退回次数、失败维度、评分变化都是 harness 自己记录的，不可能捏造）。

## 现在什么状况

Claude H-H 目前是 v0.3，MIT 开源，已在 GitHub 发布。

核心数据：
- 代码准确率从 ~50% 提升到 80%+（基于 367 个真实生产 bug 的统计）
- 8 维度评分引擎（功能正确性 33%、规格合规 18%、类型安全 14%、安全性 12%...）
- 5 个 Hook 物理约束（编辑前/后、提交前、子任务完成后、停止前）
- 支持 Python / TypeScript / Vue（Go / Rust / Java 计划中）
- 经验自学习系统（从 pipeline 退回中自动提取教训）

三个版本的演进：
- **v0.1**：解决"AI 写的代码质量不够"
- **v0.2**：解决"AI 会绕过质量检查"
- **v0.3**：解决"AI 不断重复同样的错误"

## 怎么使用

**一行安装**：
```bash
curl -fsSL https://raw.githubusercontent.com/penguinliao/claude-hh/main/install.sh | bash
```

**在项目里启用**：
```bash
cd your-project/
harness init
```

**然后正常用 Claude Code**。告诉 Claude "做 XX 功能"，Claude H-H 会自动让它走 5 阶段流水线。你只需要：
- **开头**：告诉 Claude 做什么
- **结尾**：看"全部通过"的报告，确认验收

中间的 SPEC → IMPLEMENT → REVIEW → TEST 全自动，PM 不需要介入。

## 适用场景和成本提醒

**适合你的场景**：
- 用 Claude Code 做生产级项目（不是 5 分钟原型）
- 受够了 Claude 说"修好了"但其实没修好的循环
- 产品经理 / 非技术开发者，关心代码质量但看不懂代码
- 付费客户项目、开源发布、任何"出了 bug 有代价"的场景

**不适合的场景**：
- 只写 5 分钟小玩具（pipeline 的流程开销会让你抓狂）
- 用 Cursor / Aider / Continue（目前只支持 Claude Code）
- 写 Ruby / Go / Rust（目前只支持 Python / TypeScript / Vue）

**Token 成本提醒**：
Claude H-H 会让你的 token 消耗增加 **2-3 倍**。原因是 5 阶段流水线需要多次工具调用（ruff、mypy、bandit 等）和多轮 AI 交互。这是质量和成本的取舍——如果你在意的是"快速出活"而非"可靠交付"，不要装。

**环境要求**：
- Claude Code（必须，Hook 机制依赖它）
- Python 3.9+
- macOS 或 Linux（Windows 暂不支持）

## 希望帮到你什么

我做这个东西的初心很简单：**让不懂代码的产品经理也能安心用 AI 交付生产级产品。**

AI 写代码的时代已经来了，但"AI 写的代码能不能上线"这个问题还没有好的答案。大多数人的做法是"让 AI 自己说质量够不够"——这就像让考生自己批改试卷。

Claude H-H 的答案是：**让检查者和被检查者不是同一个实体。** AI 写代码，harness 检查代码。AI 不能绕过 harness，就像你不能自己给自己的论文打分。

如果你也有类似的困扰，试试看。一行命令安装，不满意一行命令卸载。

**能绕过的规则不是规则。能验证的质量才是质量。**
