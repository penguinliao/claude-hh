# 功能正确性规则

> 复制以下内容到项目的CLAUDE.md中，AI生成代码时自动遵循

## 规格先行
- 写任何代码之前，先输出：我理解的需求 + 验收标准（当X时应该Y）+ 修改范围
- 用户确认后才开始写代码
- 复杂功能（4+文件）必须走七阶段流水线

## 模块化实现
- 复杂功能分成独立函数，每个函数有明确的输入/输出类型注解
- 每个函数写完后立即写一个简单调用验证（self-test）
- 不要一次写完所有代码再测试

## 执行验证
- 写完后必须自己运行验证，不要让用户替你跑测试
- 运行报错必须修复后再交付，不要交付有报错的代码
- 如果运行需要特定环境（数据库/API），说明前置条件

## 错误处理
- 所有后端错误必须有前端可见的提示（禁止静默catch）
- except块至少要记日志（logger.error），不要pass
- 第三方API调用必须有超时和重试
- 降级链的最后一环也要有try/except

## 交叉自查
- 交付前用"审查者"视角检查：
  □ 所有API端点是否有认证？
  □ 所有错误是否有前端可见提示？
  □ 数据库操作是否有事务保护？
  □ 前端调用的API后端都实现了吗？
  □ 新功能是否注册/调用了？（路由注册、定时任务注册、菜单入口等）
  □ 验收标准是否逐条满足？

## 完整性检查
- 新增API端点：必须有路由注册 + 认证 + 参数验证 + 错误处理 + 前端调用
- 新增数据库表：必须有CREATE TABLE IF NOT EXISTS + 迁移说明
- 新增配置项：必须有环境变量读取 + 无默认敏感值 + 文档说明
- 新增后台任务：必须用safe_task + 有日志 + 有降级
